## Swiss bar (held every PR)

Real backends via `HttpTransport`/`GraphClient` Protocol + deterministic fakes (no live SaaS
in CI; httpx-backed live paths gated). No token persistence (resolver holds only env-var
names; `call_tool` audits only kwarg key names). kg_writer reads typed inventories, never
OCSF dicts. Honest tri-state — unknown/unreadable values never produce findings. Substrate
**seal empty** the whole cycle (ADR-018 SaaS vocab was already scaffolded; D.10 is its first
consumer). ruff + mypy clean throughout.

## Honest deferrals (surfaced, not faked)

1. **`SSO_INTO` / `FEDERATED_FROM`** (SaaS → cloud-account bridge — D.10's analogue of D.6's
   IRSA bridge). Q5 named `SSO_INTO`, but its cloud-side target requires **federation-config**
   the three v0.4 connectors don't yet read. `AUTHORIZED` ships (fully data-backed); the
   cross-domain bridge is a clean follow-up (a federation-target collection step), the same
   discipline Stage 1 used deferring `HAS_ACCESS_TO`/`CAN_REACH`. **Operator decision point:
   expand connectors to collect federation targets, or leave as a v0.5 follow-up.**
2. **`SAAS_USER` enumeration** (Q5 deferred) — can be large; a follow-up slice.
3. **Salesforce / Google Workspace / HubSpot** connectors — v0.5 (Q1).
4. **M365 GraphReader reuse nuance:** SSPM reuses the identity agent's GraphReader _pattern_
   (collection seam + fake injection), not its concrete azure-identity-SDK reader, because Q3
   mandates OAuth2-from-env auth. Documented in `tools/m365.py`.
