## Q-AppSec-1 — SBOM / SCA ownership (AppSec vs D.1 Vulnerability)

**Why it's open:** #647 flags a double-assignment — PRD §7.1.5 lists Syft/Grype/
Dependency-Track under **D.1 Vulnerability**, while the competitive benchmark gives
SBOM to the **new AppSec** agent. Ownership must be resolved before either builds SBOM.

| #                          | Option                                                                                                                                            | Trade-off                                                                       |
| -------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------- |
| **(a)** ✅ **Recommended** | **AppSec owns build-time SBOM** (from source repos / CI); **D.1 owns deploy-time SBOM** (image / registry / host). Both write a shared SBOM node. | Clean build-vs-deploy split; matches the proposed ADR-014 boundary; no overlap. |
| (b)                        | **D.1 owns ALL SBOM/SCA**; AppSec does SAST + IaC + secrets only.                                                                                 | Concentrates dependency scanning in one agent; AppSec stays code-analysis-only. |
| (c)                        | **AppSec owns ALL SBOM/SCA**; D.1 does image/registry/host vuln only.                                                                             | Concentrates SBOM in the new agent; D.1 loses its PRD-assigned SCA.             |

**Team recommendation: (a)** — codify as **ADR-014** at B-1 launch.

## Q-AppSec-2 — SCM connector scope + per-tenant auth model

**Why it's open:** AppSec reads source repos; #647 notes the net-new "Repository /
CodeArtifact" nodes. Which source-control systems, and how are per-tenant credentials held?

| #                          | Option                                                                                                       | Trade-off                                                                                      |
| -------------------------- | ------------------------------------------------------------------------------------------------------------ | ---------------------------------------------------------------------------------------------- |
| **(a)** ✅ **Recommended** | **GitHub + GitLab + Bitbucket**, per-tenant via the charter Pattern-A credential resolver (OAuth app / PAT). | Covers the big three; reuses the existing credential-resolver pattern (no new auth substrate). |
| (b)                        | **GitHub-only pilot**, expand in B-2+.                                                                       | Fastest to first value; narrowest coverage.                                                    |
| (c)                        | **GitHub + GitLab** (skip Bitbucket).                                                                        | Middle ground.                                                                                 |

**Team recommendation: (a)** if the credential resolver covers SCM tokens cleanly;
fall back to (b) if per-SCM OAuth needs a new auth model (that would be a substrate
question → pause).

## Q-AppSec-3 — IaC scanner choice

**Why it's open:** PRD §7.1.7 (IaC). #647 lists checkov as a candidate (0 hits today).

| #                          | Option                   | License    | Trade-off                                                                                         |
| -------------------------- | ------------------------ | ---------- | ------------------------------------------------------------------------------------------------- |
| **(a)** ✅ **Recommended** | **Checkov**              | Apache-2.0 | Broadest coverage (Terraform / CloudFormation / K8s / ARM / Helm); subprocess pattern like Trivy. |
| (b)                        | **Trivy config scanner** | Apache-2.0 | Trivy is already a repo binary (reuse); narrower IaC rule depth than Checkov.                     |
| (c)                        | **tfsec**                | Apache-2.0 | Terraform-focused; now folded into Trivy upstream — prefer (b) if going Trivy.                    |

**Team recommendation: (a) Checkov** — breadth; permissive license (#23 clear).

## Q-AppSec-4 — Secrets-in-code scanner

**Why it's open:** PRD §7.1.6 (secrets). #647 lists gitleaks + trufflehog as candidates.

| #                          | Option                   | License         | Trade-off                                                                                                     |
| -------------------------- | ------------------------ | --------------- | ------------------------------------------------------------------------------------------------------------- |
| **(a)** ✅ **Recommended** | **gitleaks**             | **MIT**         | Permissive; fast; broad rule set; subprocess pattern.                                                         |
| (b)                        | **Trivy secret scanner** | Apache-2.0      | Already in repo (reuse); fewer rules than gitleaks.                                                           |
| (c)                        | **trufflehog**           | **AGPL-3.0** ⚠️ | ⚠️ **Pause trigger #23 (copyleft).** NOT recommended — AGPL is incompatible with the permissive-only posture. |

**Team recommendation: (a) gitleaks** (MIT) or (b) Trivy. **Avoid (c).**

## Q-AppSec-5 — SAST tool + scope

**Why it's open:** code-side detection depth. #647 lists semgrep as a candidate.

| #                          | Option                                                      | License                         | Trade-off                                                                                                                            |
| -------------------------- | ----------------------------------------------------------- | ------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------ |
| **(a)** ✅ **Recommended** | **Semgrep (OSS)** + curated ruleset, top languages first    | LGPL-2.1 (CLI)                  | Broad language coverage; subprocess use of an LGPL CLI is compatible; registry rules carry their own licenses (vet the chosen pack). |
| (b)                        | **CodeQL**                                                  | proprietary (free for OSS only) | ⚠️ commercial-use license restrictions — caution.                                                                                    |
| (c)                        | **Defer SAST** to B-2+; B-1 = SBOM/SCA + IaC + secrets only | —                               | Smaller B-1; SAST (the deepest piece) lands once the pilot proves out.                                                               |

**Team recommendation: (a) Semgrep OSS** (vet the ruleset license), or **(c) defer** if
B-1 scope should stay tight.
