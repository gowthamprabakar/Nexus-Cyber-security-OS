## 1. Decisive finding — the simulator is fully built + unit-tested, just never called in run()

The effective-perms machinery exists and is exercised by its own tests; the gap is a **single
missing call site in `run()`**, not missing code:

- **`resolve_effective_grants(listing, simulator_results) -> tuple[EffectiveGrant, ...]`**
  ([identity/tools/permission_paths.py:44-72](../../packages/agents/identity/src/identity/tools/permission_paths.py#L44-L72))
  — flattens IAM simulator decisions into `EffectiveGrant` rows (`principal_arn`, `action`,
  `resource_pattern`, `effect`, `source_policy_arns`, `is_admin`), drops implicit-denies, flags
  admin-equivalent actions.
- **`aws_iam_simulate_principal_policy(principal_arn, actions, resources=("*",), …)`**
  ([identity/tools/aws_iam.py:337-385](../../packages/agents/identity/src/identity/tools/aws_iam.py#L337-L385))
  — wraps AWS IAM `SimulatePrincipalPolicy` (batches ≤50 actions/req, runs in `asyncio.to_thread`).
  Already **charter-registered** at agent.py:74-78.
- Unit coverage in `test_permission_paths.py`; **no integration test drives it from run()**.

## 2. The exact gap — run() short-circuits to a hardcoded pattern-match

`agent.run()` fetches the IAM listing (incl. policy documents, new in v0.2 Task 6) then calls
**`_synthesize_admin_grants(listing)`**
([identity/agent.py:302-340](../../packages/agents/identity/src/identity/agent.py#L302-L340))
which **does NOT call the simulator** — it pattern-matches principals with `AdministratorAccess`
attached and fabricates a single `EffectiveGrant(action="*:*", is_admin=True)`. The in-code
comment is explicit: _"Phase 1 v0.1 derives admin grants directly from the listing's attached
policy ARNs … instead of running the IAM simulator per principal — the simulator wrapper is
exercised by its own tests and reserved for Phase 2 finer-grained scans."_

Net: the listing (the simulator's input) is **already gathered**; the wiring that feeds it to
`aws_iam_simulate_principal_policy` → `resolve_effective_grants` → normalizer is the whole job.

## 3. Inputs — mostly ready; three honest gaps

| Input                                  | Status                                                                          |
| -------------------------------------- | ------------------------------------------------------------------------------- |
| Principal ARNs (users/roles/groups)    | ✅ enumerated                                                                   |
| Customer-managed policy documents      | ✅ fetched (v0.2 Task 6)                                                        |
| Attached + group-inherited policy ARNs | ✅                                                                              |
| Inline policy **documents**            | ❌ only _names_ listed — needs `get_user/role/group_policy` bodies              |
| Resource policies (S3/KMS bucket/key)  | ❌ not fetched (Access Analyzer covers external-access slice)                   |
| SCPs / condition keys                  | ❌ Phase-1 single-account, conditions ignored (per `permission_paths.py:11-12`) |

**Azure:** `AzureAdListing` is shape-disjoint (users/groups/service-principals only) with **no
role-assignment or permission-simulation API wired** — Azure effective-perms is a separate,
larger build. AWS-first is the honest scope for A-4.

## 4. Output — refines existing OCSF 2004; no new class needed

Driven, the simulator would feed the existing **`OVERPRIVILEGE`** finding (OCSF 2004) with
real per-action/per-resource grants instead of "has AdministratorAccess attached." The schema
already carries an **unused `ADMIN_PATH` finding type** (`FindingType` enum, schemas.py:98-106;
`FINDING_ID_RE` already admits `IDENT-ADMIN-*`) for an optional privilege-chain finding. **No new
OCSF class.**

## 5. Forks for operator decision (BEFORE implementation)

### Fork A-4-1 — simulation scope (the latency/coverage knob)

- **(1a) Risk-weighted curated action set (~50-100 high-risk actions: `iam:*`, `s3:*`,
  `sts:AssumeRole`, …) — RECOMMENDED:** bounded simulator cost, highest ROI, honest first step.
- **(1b) Enumerate actions from each principal's own policies:** catches custom sets; more calls.
- **(1c) Full AWS action taxonomy (10k+):** complete but combinatorially expensive; defer.

### Fork A-4-2 — emission shape

- **(2a) Refine existing OVERPRIVILEGE (RECOMMENDED):** backward-compatible, evidence-payload-only
  change, same class/fields.
- **(2b) Add `ADMIN_PATH` privilege-chain finding:** richer for investigation; new finding type
  (schema slot already reserved).

### Fork A-4-3 — drive cadence

- **(3a) Synchronous in run() (RECOMMENDED):** matches v0.2 single-scan shape, deterministic for
  CI/eval; latency bounded by Fork-1a's curated set + per-principal timeout (already parameterized).
- **(3b) Separate `analyze-grants` CLI pass on a saved listing snapshot:** decouples latency; two-step.
- **(3c) Continuous slow-cadence (Phase-C scheduler):** background refresh; staleness tradeoff.

### Fork A-4-4 — inline-policy fetch: **(4a) skip for v0.3** (attachment-based detection already

covers most inline admins) vs **(4b) fetch inline bodies** (closes the pure-inline edge case).

## 6. Substrate (#19/#29) + effort

- **Substrate CLEAR:** `EffectiveGrant`, the simulator wrapper, and the normalizer are all
  identity-agent-local; the simulator tool is already charter-registered. **No `packages/shared` /
  `packages/charter` edit.**
- **Latency caution (not substrate):** simulating 100+ principals × batched actions can add tens
  of seconds/run — mitigate via Fork-1a curated set + CI uses a reduced set / faked simulator
  (the live lane carries the full set behind `NEXUS_LIVE_*`).
- **Realistic effort:** ~7-11 days for all forks at the recommended (a) options (core wiring
  4-6d + scope + emission refine).
