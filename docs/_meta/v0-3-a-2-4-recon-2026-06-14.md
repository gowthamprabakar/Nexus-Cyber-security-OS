## 1. Decisive finding — Trivy already _detects_ secrets; the normalizer silently _drops_ them

Trivy's secret scanner runs by default in `trivy fs` / `trivy image` / host scans. The
vulnerability normalizer already recognizes the class — `_to_ecosystem` maps Trivy's
`"secret"` → `"Secret"`
([normalizer.py:62-70](../../packages/agents/vulnerability/src/vulnerability/normalizer.py#L62-L70)).
But the per-row gate drops every secret finding:

```python
# normalizer.py:166-170
if not cve_id or not pkg_name:
    continue          # secret findings have NO VulnerabilityID → silently discarded
```

So the capability is **detected-but-discarded today**, not absent. A-2.4 is a _routing_
decision (where do these findings live), not a new scanner build.

## 2. Where secrets should land — DSPM (data-security, D.5) is the natural owner

- DSPM emits **OCSF 2003** and discriminates findings via the `DataSecurityFindingType` enum
  ([data-security/schemas.py:60-76](../../packages/agents/data-security/src/data_security/schemas.py#L60-L76)).
- Its `ClassifierLabel` enum is **explicitly designed for additive extension** — already grew
  7→14 in v0.2 (append-only so `classify()` precedence keeps prior matches byte-identical;
  [schemas.py:78-109](../../packages/agents/data-security/src/data_security/schemas.py#L78-L109)).
  It already carries credential-shaped labels (`AWS_ACCESS_KEY`, `JWT`, `GENERIC_API_TOKEN`).
- A new discriminator (`RUNTIME_SECRET_DETECTED`) + new label (`RUNTIME_CREDENTIAL`) is a
  **pure additive enum append** — same shape as the v0.2 expansion, offline eval byte-identical.
- DSPM's privacy contract (`assert_privacy_contract`, no plaintext in evidence; the WI-S8/Q6
  invariant) is exactly the guard secrets need — a credential value must NEVER appear in a
  finding; only a categorical label + privacy hash.

## 3. Routing path — two established cross-agent patterns exist (NO new bus needed)

- **(P1) Sibling-workspace read (RECOMMENDED):** D.1 writes its raw secret hits to its findings
  output; D.5 ingests via the existing `--sibling-workspace` pattern (precedent:
  `compliance/correlators/data_security_correlator.py`, investigation's `sibling_workspaces`).
  No coupling, no new dep, deterministic in CI.
- **(P2) Direct in-process call:** D.1 imports a D.5 emit helper. Tighter coupling; creates a
  D.1→D.5 package dependency edge. Lower latency, but worse blast radius.
- **(P3) `claims.>` bus — REJECTED:** ADR-012 reserves `claims.>` for _speculative_ state, and
  D.12 Curiosity is the sole producer. Secrets are **observed**, not hypotheses — wrong bus.

## 4. Forks for operator decision (BEFORE implementation + ADR-015)

### Fork A-2.4-1 — emission class

- **(1a) New DSPM discriminator on OCSF 2003 (RECOMMENDED):** additive `DataSecurityFindingType`
  - `ClassifierLabel` append; single owner for all data/credential findings; reuses the privacy
    contract. Substrate-CLEAR (data-security-local enums, byte-identical eval).
- **(1b) Keep in vulnerability, tag ecosystem="Secret":** would require relaxing `FINDING_ID_RE`
  (it encodes `CVE-…`) and emitting OCSF 2002 for non-CVE rows — breaks finding-id semantics.
  Not recommended.
- **(1c) New dedicated OCSF "Secrets" class:** high substrate cost (new class_uid, StreamSpec,
  charter/shared amendments per ADR-004). Only if the operator wants secrets as a first-class
  signal channel. Defer unless explicitly chosen.

### Fork A-2.4-2 — routing transport: **(2a) sibling-workspace (P1, RECOMMENDED)** vs **(2b) in-process call (P2)**.

### Fork A-2.4-3 — scanner source of truth

- **(3a) Reuse Trivy's built-in secret scanner (RECOMMENDED):** already shipping, Apache-2.0,
  zero new dep. Just stop dropping the rows.
- **(3b) Add gitleaks (MIT) for richer rules:** new dep + binary; only if Trivy's ruleset proves
  thin. **Avoid trufflehog — AGPL (#23 license pause-trigger).**
