# F.7 v0.2 — D.7 Investigation lifecycle events on `events.>`

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` (recommended) or `superpowers:executing-plans` to implement this plan task-by-task. Pause for review after each numbered task.

**Goal.** Turn the F.7 v0.1 substrate from "live but unused by any agent" into "exercised by one real agent" — by migrating **D.7 Investigation's lifecycle events** (Stage-start / Stage-complete / Investigation-complete) onto the `events.>` bus. This validates that the foundation works with ONE real consumer of the substrate before any other agent migrates. D.7 is the first Phase-1b agent; it already uses the F.5/F.6 substrate end-to-end, and it now adds the F.7 substrate as a fourth substrate it depends on.

**Hard scope boundary** (stated up-front because the original A.1 three-tiers-into-one collapse is the cautionary tale this plan is written against — and because F.7 v0.1's plan only held its boundary because it was named verbatim):

**F.7 v0.2 = D.7 lifecycle events onto `events.>` ONLY.**

Specifically out of scope, each named so it doesn't drift in mid-execution:

- **NOT** Cloud-Posture migration. Cloud-Posture publishing findings to `findings.>` is **F.7 v0.3+ (Cloud-Posture's own plan, not started)**.
- **NOT** D.5 (Network Threat) migration. **F.7 v0.3+, separate plan, not started.**
- **NOT** D.6 (Runtime Threat) migration. **F.7 v0.3+, separate plan, not started.**
- **NOT** D.7's `find_related_findings` sibling-reads onto `findings.>`. D.7 keeps reading sibling workspaces via the filesystem path in v0.2. The bus-mediated consume side lands only when the sibling agents (Cloud-Posture / D.5 / D.6 etc.) publish to `findings.>` — which is their v0.3+ scope, not D.7's.
- **NOT** D.7's incident-report finding (`incident_report.json` OCSF 2005) onto `findings.>`. D.7 keeps writing it to the charter workspace in v0.2. The finding-publish migration is a later v0.x.
- **NOT** any new D.7 detection capability — no new sub-investigations, no new stage logic, no new MITRE techniques, no LLM-prompt changes. Pure transport migration of lifecycle events D.7 already emits to logs.
- **NOT** edge-plane leaf-node migration. Still **E.1 + E.2 scope.**

Collapsing "D.7 lifecycle events" + "D.7 finding-publish" + "D.7 sibling-read migration" into one plan would repeat the A.1 three-tiers-into-one failure mode. **One agent, one direction (publish-side), one event-class (lifecycle events).** Three knobs locked.

## Strategic context

After F.7 v0.1 closed (PR #22, commit `cf3a1fd`), the substrate is empirically demonstrated against a real broker. The [F.7 v0.1 verification record](../../_meta/f-7-v0-1-verification-2026-05-17.md) names D.7 v0.2 as the immediate next-plan gate. The platform-line criterion: **F.7 v0.2 validates that the substrate works with one real publisher before five more agents stack on top of it.** If F.7 v0.2 finds a sharp edge — a contract D.7 needs that v0.1 doesn't provide, a behaviour difference between mocked and live, a back-pressure-under-real-load surprise — the v0.3+ agent-migration plans inherit a flat-tested foundation rather than discovering the same edges in parallel.

Why D.7 first (and not Cloud-Posture or another detect agent):

1. **D.7's lifecycle events are operator-valuable today.** S.1 Console (future) wants real-time agent activity; A.4 Meta-Harness (future) wants trace aggregation. D.7's "Stage X started / completed" stream is the smallest meaningful artefact either consumer needs.
2. **D.7 has the lightest scope-collapse risk.** D.7 doesn't _produce_ findings the way Cloud-Posture does — it produces `incident_report.json` (OCSF 2005) as a synthesis output, but that's not the v0.2 surface. Migrating D.7 lifecycle events does not touch any other agent's path.
3. **D.7 already uses three F.x substrates (F.1 / F.5 / F.6).** Adding F.7 as a fourth substrate dependency is an additive code path, not an architecture change.
4. **D.7's existing eval suite (10 cases) gives a back-compat gate.** All 10 must continue to pass with the bus-publish flag OFF (no regression) AND ON (the new path doesn't change outcomes).

## Scope

**In v0.2 (load-bearing):**

1. **D.7 emits lifecycle events to `events.>`.** Three event types per investigation: `investigation.started` (Stage 1 SCOPE entry), `investigation.completed` (Stage 6 HANDOFF success), `investigation.failed` (Stage 6 HANDOFF failure or upstream stage failure). Subject pattern: `events.tenant.<tenant_id>.investigation.<event_type>` via the existing `shared.fabric.subjects.events_subject()` helper.
2. **The migration is opt-in and reversible.** `--publish-events-to-bus` CLI flag (default off) + `NEXUS_FABRIC_PUBLISH=1` env var. When off, D.7's behaviour is byte-identical to F.7 v0.1's state.
3. **Publish failures are non-fatal.** A `FabricConnectionError`, timeout, or NATS write error during lifecycle publishing logs at WARNING level + is recorded in the F.6 audit chain as `investigation.bus_publish.failure`, but does NOT fail the investigation. D.7's "filesystem artifacts are the contract" guarantee is preserved.
4. **`NEXUS_LIVE_NATS=1` integration test proves the migration end-to-end.** A real D.7 run against a real broker; a test-side subscriber on `events.tenant.<tid>.investigation.>` captures the published events; assertions on payload + `Nexus-Correlation-Id` header. Same A.1-live-cluster-grade discipline as F.7 v0.1 Task 6.
5. **All 10 D.7 eval cases pass with the flag OFF AND with the flag ON.** Back-compat preserved (off) and additive-only proven (on: outcomes unchanged; only the bus-publish side-effect differs).

**Explicitly out of v0.2 (each pinned in the hard scope boundary):**

- D.7's `find_related_findings` sibling-reads onto `findings.>`. Filesystem path stays.
- D.7's `incident_report.json` (OCSF 2005) publishing to `findings.>`. Charter workspace write stays.
- Cloud-Posture / D.5 / D.6 finding-publish migrations onto `findings.>`. Each is its own v0.3+ plan.
- Any new D.7 detection capability. No new sub-investigations, no new stages, no new MITRE techniques.
- KMS-signed `audit.>`, tenant ACL via NATS auth tokens, protobuf schemas on the 4 non-findings streams. All v0.x hardening.
- Edge-plane leaf-node. E.1 + E.2.

## Reversibility and rollback

The v0.2 migration is **additive and reversible** by design:

- **Default-off flag.** `--publish-events-to-bus` defaults to false. Operators not opting in see D.7's v0.1 behaviour exactly. The bus is exercised by v0.2 only when the flag is explicitly set.
- **Filesystem path unchanged.** D.7's Stage-6 HANDOFF continues to write the 4 charter-workspace artifacts (`incident_report.json` / `timeline.json` / `hypotheses.md` / `containment_plan.yaml`). Bus publishing is a side-effect, not a replacement.
- **Publish failures non-fatal.** If the bus is unreachable while the flag is on, D.7 logs WARNING + emits `investigation.bus_publish.failure` to the F.6 chain, then continues. Investigation outcomes remain reliable.
- **Eval-suite gate.** All 10 eval cases must pass with the flag OFF (proves no regression) AND ON (proves additive-only). A migration that changes any eval outcome is rejected — outcomes are part of D.7's behavioural contract.

**Rollback path** (if v0.2 introduces a defect found post-deploy):

1. Set `--publish-events-to-bus=false` (or unset `NEXUS_FABRIC_PUBLISH`) on operator runs. D.7 reverts to v0.1 behaviour with zero code changes.
2. If the defect is in the always-evaluated flag-check path itself (unlikely — the check is one `if` statement), revert the v0.2 PRs in order; no migrations to undo because no agent depends on D.7 publishing to the bus today.
3. The verification record will document the rollback path explicitly.

Same principle as A.1 v0.1.1's "additive code paths, prior behaviour preserved" (per [A.1 v0.1.1's verification record](../../_meta/a1-v0-1-1-verification-2026-05-17.md#breaking-change-note-for-execute-mode-operators)).

## Version-dependency check

D.7's v0.2 migration uses **only the v0.1 `JetStreamClient` public API** — `connect()` / `ensure_streams()` / `publish()` / `close()`. The publish uses `Nexus-Correlation-Id` header propagation (NATS 2.0+) and the same `events.>` stream `StreamConfig` shape v0.1 declared (`max_age` / `discard` / `subjects` — all NATS 2.10+).

**No NATS feature exceeding the `nats:2.10-alpine` canonical baseline is introduced.** The compose-image pin (`nats:2.10-alpine`) remains the correct production target.

**The F.7 v0.1 §6 version-deviation note carries forward** verbatim into the v0.2 verification record. The v0.2 live proof will run against whatever broker the v0.2 implementer has available (Docker `nats:2.10-alpine` or brew `nats-server v2.14.0`); both are protocol-equivalent for the surface v0.2 exercises (publish with headers, subscribe with push consumer, `events.>` stream config). The v0.2 verification record's "Permanent documented limitation" section is the same shape as v0.1's §6 with a sentence appended: "v0.2 added no new NATS-version dependency; the `v2.14.0`-vs-`2.10-alpine` deviation stated in v0.1 §6 still applies and still requires no production change."

## Compatibility contract

F.7 v0.2 preserves F.7 v0.1's public surface exactly:

- **`JetStreamClient.*` method signatures unchanged.** Existing callers continue to work without source changes.
- **`CORRELATION_ID_HEADER` value unchanged** (`"Nexus-Correlation-Id"`). Consumers (today's tests; tomorrow's S.1 / A.4) read the same header name.
- **`StreamSpec` shape unchanged.** The 5 ADR-004 streams keep their v0.1 retention / subjects / discard configuration.
- **`shared.fabric.*` **init** exports** — additive only (no symbol renamed or removed).

F.7 v0.2 also preserves D.7's existing surface:

- **D.7 CLI subcommands unchanged.** Existing `investigation-agent run` / `triage` / `eval` invocations continue to work without flag changes.
- **D.7's 10 eval cases pass with byte-identical outcomes** when the flag is off (back-compat gate, Task 6).
- **D.7's 4 charter-workspace artifacts** (`incident_report.json` / `timeline.json` / `hypotheses.md` / `containment_plan.yaml`) are still written. The bus side is additive.
- **D.7's `find_related_findings` sibling-workspace reads** (filesystem path) are untouched. The cross-agent consume side does not migrate in v0.2.

This contract becomes the v0.3 starting point.
