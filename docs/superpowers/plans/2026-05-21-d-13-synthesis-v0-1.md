# D.13 — Synthesis Agent v0.1 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` (recommended) or `superpowers:executing-plans` to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking. Pause for review after each numbered task.

**Goal:** Ship the **Synthesis Agent** (`packages/agents/synthesis/`) — the **fourth of the 7 unbuilt agents** under the [Path-B-breadth-first operating rule](../sketches/2026-05-20-agent-version-roadmaps.md) (2026-05-20) and the **fourteenth under [ADR-007](../../_meta/decisions/ADR-007-cloud-posture-as-reference-agent.md)** (F.3 / D.1 / D.2 / D.3 / F.6 / D.7 / D.4 / multi-cloud-posture / k8s-posture / A.1 / D.5 / D.8 / D.6 / **D.13**). Customer-facing narration: synthesizes findings + investigations + compliance reports into human-readable LLM-narrated summaries. **The first agent that calls the LLM in its hot path** (prior agents plumb `llm_provider` but never invoke it).

**Scope (v0.1, locked per Path-B-breadth-first rule + sketch §5).** Two narrative artifacts: an `executive_summary.md` (1-paragraph C-suite digest) + a longer `narrative.md` (sectioned per-finding-class). Three sibling sources: **D.7 Investigation**, **D.6 Compliance**, **F.3 Cloud Posture** workspaces (read-only, operator-pinned via flags). Two LLM calls per run (outline + narration), both seeded `temperature=0.0`. Stub-LLM eval harness keeps the eval suite deterministic + offline; live-LLM smoke test gated by `NEXUS_LIVE_LLM=1`. No OCSF emit in v0.1 (deferred to v0.2 once we lock a `class_uid` choice).

**Strategic role.** Fourth agent in the breadth-first cadence. Closes the **customer-facing narration loop**: the prior 12 agents produce structured findings (OCSF dicts); D.13 turns them into prose a CISO can read. The LLM-narration pattern is **read-only over sibling workspaces** — D.13 reads but never writes back. Mirrors D.7's sibling-workspace-pattern + D.6's cross-source aggregation shape. **No charter-level substrate work expected** — uses existing `charter.llm_adapter` (ADR-006) + sibling-workspace reads.

**Q1 (resolve up-front).** Output shape — OCSF Detection/Compliance Finding or markdown reports?

**Resolution: markdown reports, no OCSF emit in v0.1.** D.13 outputs `narrative.md` + `executive_summary.md` to the charter workspace. The F.6 audit chain carries a `synthesis_produced` event for traceability. OCSF emit is deferred to v0.2 (requires a `class_uid` choice — 1006 "Web Resources Activity"? new class? — that's its own ADR). v0.1 keeps the contract minimal: operator reads markdown, downstream consumers ingest via the F.6 audit chain.

**Q2 (resolve up-front).** Sibling sources — which workspaces?

**Resolution: three operator-pinned workspaces** via `--investigation-workspace`, `--compliance-workspace`, `--cloud-posture-workspace`. D.7 Investigation conclusions are the narrative spine; D.6 Compliance posture is the compliance section; F.3 Cloud Posture is the technical-details fallback when D.7 isn't pinned. **D.1 / D.4 / D.5 / D.8 land as findings consumed by D.6 / D.7** — D.13 reads them transitively, not directly. **D.12 Curiosity hypothesis narration deferred to v0.2** per the sketch (Curiosity itself isn't shipped yet).

**Q3 (resolve up-front).** LLM provider — direct adapter or new substrate?

**Resolution: reuse `charter.llm_adapter.LLMProvider`** (ADR-006). Stub provider for tests via a local `make_stub_provider(canned_responses)` helper in the test scaffold (not exported substrate). v0.1 ships with the default Anthropic adapter; operators stub for eval cases. No new substrate.

**Q4 (resolve up-front).** LLM call structure — single call or multi-step?

**Resolution: two-call structure.** Stage 3 NARRATE issues:

1. **Outline call** — given the structured context bundle from Stage 2 ENRICH, produce a structured outline (JSON dict: section list + per-section key-findings-to-cite). Validated against a pydantic schema; bad outputs raise a typed `OutlineCallError` and the agent emits a fallback "synthesis failed: see findings.json for raw data" narrative.
2. **Narration call** — given the validated outline, produce the full markdown narrative. Section-by-section iteration (one inner call per section in the outline) so token usage is predictable and a single-section failure doesn't poison the whole document.

Both calls use `temperature=0.0` and pin the model via `envelope.model_pin`. Two prompt templates per call type (outline / narration / executive_summary) → 3 templates total, each ~50 lines of markdown, loaded via `importlib.resources`.

**Q5 (resolve up-front).** Tenancy — single-tenant or multi-tenant in v0.1?

**Resolution: single-tenant** (same as D.5 / D.6 / D.8). Multi-tenant production blocks on the future SET LOCAL `$1` tenant-RLS substrate-fix plan.

**Q6 (resolve up-front).** Q6 contract from D.5 — how does D.13 honour the no-classifier-substring invariant when the LLM is in the loop?

**Resolution: two layers of defence.**

1. **Context bundle (Stage 2 ENRICH) is structured-fields-only.** When D.13 reads D.5 findings transitively (via D.6's aggregated compliance findings or D.7's investigation conclusions), the context bundle carries only:
   - Severity counts.
   - Top-N finding-ids + their titles + control mappings.
   - D.5 `classifier_labels_found` array (the _labels_, not the matched substrings).
   - NEVER raw bucket-object names, key fragments, or `evidence.bucket_objects[].matched_text`-shaped fields.
2. **Reviewer (Stage 4 REVIEW)** runs a deterministic regex pass over the rendered narrative for known classifier patterns (SSN-shape, credit-card-shape, AWS-access-key-shape, etc.). If the LLM hallucinates a "looks plausible" classifier substring, the reviewer rejects the narrative and the agent re-runs the narration call with a `q6_violation_retry` hint in the prompt. Eval case 007 (`no_classifier_substrings`) is the regression probe.

The narrative may _reference_ a finding's `title` + `control_id` + `severity` + `affected_resource.arn` (those are public-shape; auditors expect them) but never the matched-text fragment that triggered the finding. **No PII leaks via LLM hallucination** is the WI-2 acceptance criterion.

## Architecture

```
ExecutionContract (signed)
        │
        ▼
┌──────────────────────────────────────────────────────────────────┐
│ Synthesis Agent driver                                           │
│                                                                  │
│  Stage 1: INGEST     — 3 sibling-workspace reads (TaskGroup)     │
│  Stage 2: ENRICH     — build structured context bundle for LLM   │
│  Stage 3: NARRATE    — 2 LLM calls (outline + narration)         │
│  Stage 4: REVIEW     — deterministic narrative validator         │
│                        + Q6 classifier-substring guard           │
│  Stage 5: SUMMARIZE  — assemble narrative.md + exec_summary.md   │
│  Stage 6: HANDOFF    — write markdown files to charter workspace │
└─────────┬────────────────────────────────────────────────────────┘
          │
          ▼
┌──────────────────────────────────────────────────────────────────┐
│ Tools (per-stage)                                                │
│  read_sibling_workspaces    ─→ D.7 / D.6 / F.3 findings.json     │
│  build_context_bundle       ─→ structured LLM-input dict         │
│  prompts/outline.md         ─→ Stage-3 outline-call template     │
│  prompts/narration.md       ─→ Stage-3 narration-call template   │
│  prompts/exec_summary.md    ─→ Stage-3 exec-summary template     │
│  narrator.narrate           ─→ 2-call LLM orchestration          │
│  reviewer.review            ─→ Q6 substring guard + shape check  │
│  kg_writer.upsert_synthesis ─→ SemanticStore (entity_type="     │
│                                synthesis_report")                │
└──────────────────────────────────────────────────────────────────┘
```

**Tech stack.** Python 3.12 · BSL 1.1 · charter.llm_adapter (ADR-006) for LLM calls · pydantic 2.9 · click 8 · charter.nlah_loader (ADR-007 v1.2). No external network dependencies beyond the LLM call itself (stubbed in eval suite per Q3).

**Depends on:** F.3 Cloud Posture (shipped) + D.6 Compliance (shipped 2026-05-21) + D.7 Investigation (shipped). **No** v0.1 dep on D.5 / D.8 / D.12 (those flow transitively via D.6 / D.7).
