# D.12 — Curiosity Agent v0.1 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` (recommended) or `superpowers:executing-plans` to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking. Pause for review after each numbered task.

**Goal:** Ship the **Curiosity Agent** (`packages/agents/curiosity/`) — the **fifth of the 7 unbuilt agents** under the [Path-B-breadth-first operating rule](../sketches/2026-05-20-agent-version-roadmaps.md) (2026-05-20) and the **fifteenth under [ADR-007](../../_meta/decisions/ADR-007-cloud-posture-as-reference-agent.md)** (F.3 / D.1 / D.2 / D.3 / F.6 / D.7 / D.4 / multi-cloud-posture / k8s-posture / A.1 / D.5 / D.8 / D.6 / D.13 / **D.12**). **The first generative agent in the fleet** — emits hypotheses about what might be under-scanned, not findings about what was scanned. **The first agent that publishes to the new `claims.>` substrate** introduced by [ADR-012](../../_meta/decisions/ADR-012-claims-subject-namespace.md).

**Scope (v0.1, locked per Path-B-breadth-first rule + sketch §4 + the 2026-05-21 ADR-012 unblock).** Three emit directions per run: a `SemanticStore` entity (entity_type="hypothesis"; persistent), a `claims.>` fabric publish (subject `claims.tenant.<tid>.agent.curiosity`; consumed by D.7/D.5/D.8 in their v0.2 plans), and a `hypotheses.md` workspace markdown for operator review. One deterministic gap detector (**region-gap** — regions with ≥10 assets but zero findings in the last 30 days). Single LLM call per detected gap (max 5 per run; structured JSON output). Stub-LLM eval harness + live-LLM smoke gated by `NEXUS_LIVE_LLM=1`. Single-tenant `semantic_store=None` opt-in default. **No probe-directive consumer integration in v0.1** — D.7/D.5/D.8 wire-up lands in those agents' v0.2 plans.

**Strategic role.** Fifth agent in the breadth-first cadence. Closes the **proactive-exploration loop**: D-track agents react to scan inputs; D.12 looks at the aggregate output and asks "what might we be missing?" The LLM-driven hypothesis pattern mirrors D.7's, but the direction is opposite — D.7 explains observed events; D.12 proposes targets to look at. **First producer on the `claims.>` substrate**; A.4 Meta-Harness will be the second once it ships.

**Q1 (resolve up-front).** Output shape — KG entity only, fabric publish only, or both?

**Resolution: three directions** (defence-in-depth + multiple consumer shapes).

1. **`SemanticStore` entity** (`entity_type="hypothesis"`) — persistent record for cross-run reference + future A.4 Meta-Harness aggregation. External_id = `<customer_id>:<run_id>:<hypothesis_idx>`.
2. **`claims.>` fabric publish** — dogfoods the substrate ADR-012 just shipped. Subject: `claims.tenant.<tid>.agent.curiosity`. Stream: `CLAIMS_STREAM`. D.12 is the first publisher; consumers (D.7/D.5/D.8) wire up in their own v0.2 plans, not here.
3. **`hypotheses.md` workspace markdown** — operator-readable artifact (mirrors D.13's narrative.md pattern). Lists each hypothesis + the probe directive in plain prose.

Wire format on `claims.>` (deferred from ADR-012, resolved here): **lightweight `nexus_claim` envelope** (NOT OCSF). A claim is a JSON object with `claim_id` (ULID) + `customer_id` + `agent_id="curiosity"` + `hypothesis` (1-2 sentence statement) + `rationale` (3-5 sentence justification) + `probe_directive` (structured) + `emitted_at`. **No OCSF class_uid** — claims are not findings; an OCSF stretch would be premature.

**Q2 (resolve up-front).** Generation pattern — deterministic, LLM-driven, or hybrid?

**Resolution: deterministic-trigger + LLM-hypothesis.** Two-stage:

- **Trigger** (deterministic): SemanticStore query identifies a coverage gap. If no gap, the run emits zero hypotheses + a `no-coverage-gaps` claim. No LLM call.
- **Hypothesis** (LLM-driven): given the gap + recent threat-intel context, single LLM call returns a structured JSON payload (`{"hypotheses": [...]}`). One LLM call per run, max 5 hypotheses emitted per run (budget cap). Pydantic-validated; bad output raises a typed `HypothesisCallError` and the driver emits a fallback "hypothesis generation failed; see logs" claim.

Both stages pin `temperature=0.0`. Model pinned at `claude-haiku-4-5-20251001` by default (operator override via `--model-pin`).

**Q3 (resolve up-front).** Coverage-gap detector — which gap shape ships in v0.1?

**Resolution: region-gap only.** ONE detector in v0.1:

- **Region gap**: regions where `SemanticStore.list_entities(entity_type="aws_account_region")` returns rows with `asset_count >= 10` but zero findings from F.3/D.5/D.6 in the last 30 days (queried via `SemanticStore` finding-aggregate entities or a tiny helper). The hypothesis statement is "scan region X — coverage gap detected"; the probe directive targets D.7 (Investigation) or D.5 (Data Security) depending on the gap's character.

**Deferred to v0.2 / v0.3+:**

- **v0.2:** asset-type gap, time-window gap, severity-distribution gap, classifier-label gap, control-coverage gap (per CIS framework).
- **v0.3+:** cross-customer baseline drift; ML-driven anomaly detection; closed-loop ("hypothesis verified by D.7 → mark claim resolved").

**Q4 (resolve up-front).** Probe directive shape — workspace-readable JSON or fabric-only?

**Resolution: structured dict written to BOTH the workspace JSON and the `claims.>` fabric payload.** Each directive carries:

- `target_agent`: enum `"investigation"` | `"data_security"` | `"threat_intel"`.
- `target_resource_arn` OR `target_finding_id`: the thing to look at (exactly one of the two).
- `action`: enum `"scan"` | `"investigate"` | `"enrich"`.
- `rationale_ref`: link to the parent hypothesis `claim_id`.

v0.1 writes these to `probe_directives.json` in the workspace + nests the same payload inside each `claims.>` message. D.7/D.5/D.8 consumer integration lands in their v0.2 plans — D.12 is producer-only here.

**Q5 (resolve up-front).** Tenancy — single-tenant or multi-tenant in v0.1?

**Resolution: single-tenant** (same as every prior agent). Multi-tenant production blocks on the future SET LOCAL `$1` tenant-RLS substrate-fix plan. **D.12 explicitly does NOT do cross-tenant analysis** (privacy violation per the sketch §4).

**Q6 (resolve up-front).** Q6 invariant — how does D.12 honour the no-classifier-substring posture when LLM-generating hypothesis text?

**Resolution: reuse D.13's reviewer.** D.12 reads SemanticStore which may carry D.5-derived classifier labels (per the ADR-012 + D.13 Q6 contract, labels not values). The LLM-generated hypothesis text MUST NOT reintroduce classifier-shaped substrings. Task 7 ships D.12's reviewer as a thin wrapper over D.13's `synthesis.reviewer._scan_classifier_labels` (refactored to a shared helper if reuse-via-import is awkward). Eval case 09 (`q6_no_classifier_substring_in_hypothesis`) is the WI-2 regression probe.

**Subscriber-ACL fence** (from ADR-012): D.12 itself is NOT in the forbidden-subscriber set — it freely subscribes to `claims.>` if v0.2+ adds a Curiosity-consumes-Curiosity loop. A.1 Remediation remains the only v0.1 entry in `_FORBIDDEN_SUBSCRIPTIONS`.

## Risks

| Risk                                                                                             | Mitigation                                                                                                                                                                                                                                                                            |
| ------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Region-gap detector signal is too coarse — emits noisy "scan region X" claims for every customer | v0.1 threshold (≥10 assets + zero findings in 30d) is intentionally conservative. Eval case `single_region_gap` calibrates the floor. Future v0.2 adds asset-type + classifier-label gap detectors for finer signals.                                                                 |
| LLM hallucinates claim text that introduces Q6 classifier-shaped substrings                      | Reviewer (Task 7) regex-guards the rendered hypothesis text + probe directive rationale. On Q6 violation: re-run hypothesize() with `q6_violation_retry_hint=True`. Eval case 05 is the regression probe. Retry budget=1 (matches D.13).                                              |
| `claims.>` fabric publish fails (NATS broker down) → run aborts                                  | `js_client=None` opt-in default: when no client is configured, PUBLISH stage is a logged no-op (matches D.6/D.8/D.13's `semantic_store=None` pattern). Production runs need explicit NATS wiring; v0.1 default is graceful skip.                                                      |
| Hypothesis count blows past budget                                                               | Hypothesizer truncates LLM output to max 5 hypotheses per run + logs the trim. Eval case `max_5_hypotheses_cap` is the regression probe. Future v0.2 may revisit the cap.                                                                                                             |
| Multi-tenant SemanticStore query violates privacy posture                                        | v0.1 explicitly single-tenant. Tools/queries scope to `customer_id` from contract; no cross-customer aggregation. Test guard in Task 3: queries error if `customer_id` is None.                                                                                                       |
| Probe directives reference finding IDs that don't exist (drift)                                  | Task 4's coverage_gap_detector materializes the gap from real SemanticStore entities, but the LLM may stitch a fabricated `target_finding_id` into the probe directive. Task 7's reviewer cross-checks `target_finding_id` against the input gaps. On miss: drop the directive + log. |
| A.1 Remediation consumes claims (autonomous-action safety)                                       | Already mitigated by ADR-012's `_FORBIDDEN_SUBSCRIPTIONS` substrate fence — A.1 cannot subscribe to `claims.>`. WI-4 is verification via the existing fabric-client tests; D.12 inherits the guarantee without new tests.                                                             |
| F.7 substrate (`claims.>` stream + subject builder) drifts from ADR-012                          | F.7 v0.1's `ensure_streams()` iterates `ALL_STREAMS` declaratively. D.12 v0.1's `claims_publisher` uses `CLAIMS_STREAM` + `claims_subject` from the existing `shared.fabric` exports. Any future ADR-004/012 drift surfaces at integration test time, not at D.12 boundary.           |
| Single-tenant SemanticStore writes block on the SET LOCAL `$1` substrate bug                     | v0.1 single-tenant only; `semantic_store=None` default. Multi-tenant production blocks on the future tenant-RLS substrate-fix plan. Confirmed in `feedback_path_b_breadth_first.md`.                                                                                                  |
